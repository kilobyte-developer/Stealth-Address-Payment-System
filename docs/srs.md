# Software Requirements Specification
## Stealth Address Payment System

**Version:** 1.0  
**Date:** March 2026  
**Status:** Hackathon Draft

---

## 1. Introduction

### 1.1 Purpose
This document defines the functional and non-functional requirements for the Stealth Address Payment System — a privacy-preserving payment layer built on top of BitGo institutional wallets.

### 1.2 Scope
The system enables:
- Generation of unlinkable one-time payment addresses from a published stealth address
- Sending Bitcoin to a stealth address without revealing the receiver's identity on-chain
- Automated scanning of blockchain transactions to detect incoming payments
- Spending of detected funds via derived private keys

The system does **not** aim to replace BitGo's existing security model; it adds a privacy layer on top.

### 1.3 Definitions

| Term | Definition |
|---|---|
| **Stealth Address** | A publicly shareable pair `(A, B)` of public keys from which one-time addresses can be derived |
| **One-Time Address (OTA)** | A unique per-transaction address `P = S·G + B` that cannot be linked to the receiver's stealth address without the private view key |
| **Ephemeral Key** | Random key pair `(r, R=r·G)` generated by sender per transaction; `R` is published in the transaction |
| **Shared Secret** | `S = H(r·A) = H(a·R)` — computed independently by both parties via ECDH |
| **View Key** | Private key `a` used only to detect (not spend) incoming payments |
| **Spend Key** | Private key `b` used to derive the final private key `x = S + b` for spending |
| **Scanner** | Background service that watches the blockchain for transactions matching the receiver's view key |

### 1.4 References
- BitGo API Documentation: https://app.bitgo.com/docs
- Monero Research Lab — Stealth Addresses
- BIP 47 (Payment Codes) — related prior art
- `@noble/secp256k1` library documentation

## 2. Overall Description

### 2.1 Product Perspective
This system is an independent application that integrates with BitGo's API as its wallet and transaction broadcasting layer. It does not modify BitGo internals.

```
[User] ↔ [Web App] ↔ [REST API] ↔ [BitGo Cloud]
                             ↔ [Scanner Daemon]
                             ↔ [Database (Prisma)]
```

### 2.2 User Classes

| Role | Description |
|---|---|
| **Receiver** | A user who generates a stealth address and publishes it to receive private payments |
| **Sender** | A user who has a receiver's stealth address and wants to pay them privately |
| **Admin** | Developer/operator who manages the system |

### 2.3 Operating Environment
- Web browsers: Chrome 120+, Firefox 120+, Safari 17+
- Node.js 20 LTS (backend)
- Bitcoin Testnet (development) / Bitcoin Mainnet (production)
- BitGo Express or BitGo Cloud API

### 2.4 Assumptions & Dependencies
- BitGo access tokens are pre-provisioned
- Network connectivity to BitGo API is always available
- The receiver's scanner is always running (or run on-demand)

## 3. Functional Requirements

### FR-01: User Registration & Authentication
| ID | Requirement |
|---|---|
| FR-01-1 | Users can register with email and password |
| FR-01-2 | Users authenticate via JWT tokens (24h expiry) |
| FR-01-3 | Tokens are refreshable without re-login |

### FR-02: Wallet Management
| ID | Requirement |
|---|---|
| FR-02-1 | Users can create a new BitGo wallet through the system |
| FR-02-2 | Users can list all their registered wallets |
| FR-02-3 | Users can view wallet balance (confirmed + unconfirmed) |
| FR-02-4 | Each wallet is associated with a stealth key pair upon creation |

### FR-03: Stealth Address Generation (Receiver)
| ID | Requirement |
|---|---|
| FR-03-1 | System generates a deterministic view key pair `(a, A)` per wallet |
| FR-03-2 | System generates a deterministic spend key pair `(b, B)` per wallet |
| FR-03-3 | The stealth address `(A, B)` is exportable as a hex string and QR code |
| FR-03-4 | Private keys `a` and `b` are stored encrypted in the database |

### FR-04: One-Time Address Derivation (Sender)
| ID | Requirement |
|---|---|
| FR-04-1 | Given a receiver's stealth address `(A, B)`, the system generates a fresh random `r` |
| FR-04-2 | System computes `R = r·G` and `S = H(r·A)` and `P = S·G + B` |
| FR-04-3 | The derived address `P` and ephemeral key `R` are returned to the caller |
| FR-04-4 | Each call to the derivation endpoint MUST produce a different `r` (verified) |

### FR-05: Send Payment
| ID | Requirement |
|---|---|
| FR-05-1 | User provides: sender wallet ID, receiver stealth address, amount (sats) |
| FR-05-2 | System derives the one-time address `P` and constructs a BitGo transaction |
| FR-05-3 | Ephemeral key `R` is embedded in the transaction (OP_RETURN or tx metadata) |
| FR-05-4 | Transaction is broadcast via BitGo API |
| FR-05-5 | Transaction record is stored in DB with status `pending` |
| FR-05-6 | Transaction status is updated to `confirmed` when block confirmation is received |

### FR-06: Blockchain Scanning (Receiver)
| ID | Requirement |
|---|---|
| FR-06-1 | Scanner fetches new transactions at a configurable interval (default 30s) |
| FR-06-2 | For each transaction, scanner extracts ephemeral key `R` from metadata |
| FR-06-3 | Scanner computes `S' = H(a·R)` and `P' = S'·G + B` |
| FR-06-4 | If `P'` matches any output address, payment is recorded as `DetectedPayment` |
| FR-06-5 | Receiver is notified (in-app notification / webhook) of detected payment |
| FR-06-6 | Manual scan can be triggered via API or UI |

### FR-07: Spend Detected Payment
| ID | Requirement |
|---|---|
| FR-07-1 | Receiver can initiate spending of a detected payment |
| FR-07-2 | System derives private key `x = S + b` for the one-time address `P` |
| FR-07-3 | System builds and broadcasts a transaction spending from `P` |
| FR-07-4 | Spent payments are marked `spentAt` in DB |

### FR-08: Transaction History
| ID | Requirement |
|---|---|
| FR-08-1 | Users can view all sent transactions with status and txHash |
| FR-08-2 | Users can view all detected incoming payments |
| FR-08-3 | Transaction history is paginated (20 items per page) |

## 4. Non-Functional Requirements

### NFR-01: Security
| ID | Requirement |
|---|---|
| NFR-01-1 | Private keys `a` and `b` must be encrypted at rest (AES-256-GCM) |
| NFR-01-2 | API must reject requests without valid JWT |
| NFR-01-3 | Ephemeral key `r` is never stored; only derived outputs are persisted |
| NFR-01-4 | All communication over HTTPS (TLS 1.3) in production |
| NFR-01-5 | No sensitive data logged (keys, seeds, raw private key bytes) |

### NFR-02: Performance
| ID | Requirement |
|---|---|
| NFR-02-1 | API response time < 500ms for non-cryptographic endpoints (p95) |
| NFR-02-2 | Stealth derivation < 100ms per transaction |
| NFR-02-3 | Scanner processes up to 100 transactions per scan cycle |

### NFR-03: Reliability
| ID | Requirement |
|---|---|
| NFR-03-1 | Scanner must resume from last processed block after restart |
| NFR-03-2 | Failed BitGo API calls are retried up to 3 times with exponential backoff |
| NFR-03-3 | DB transactions are used for multi-step operations to ensure consistency |

### NFR-04: Usability
| ID | Requirement |
|---|---|
| NFR-04-1 | Stealth address is displayable as a QR code for easy sharing |
| NFR-04-2 | Error messages are human-readable, not raw error codes |
| NFR-04-3 | UI is responsive and works on mobile browsers |

### NFR-05: Maintainability
| ID | Requirement |
|---|---|
| NFR-05-1 | `packages/stealth-crypto` is zero-dependency on BitGo (testable in isolation) |
| NFR-05-2 | All environment config via `.env` — no hardcoded credentials |
| NFR-05-3 | Turborepo build pipeline caches correctly for fast rebuilds |

## 5. External Interface Requirements

### 5.1 BitGo API
- **Auth:** Bearer token in `Authorization` header
- **Wallet creation:** `POST /api/v2/tbtc/wallet` (testnet BTC)
- **Transaction build+send:** `POST /api/v2/tbtc/wallet/:id/sendcoins`
- **Transaction fetch:** `GET /api/v2/tbtc/wallet/:id/transfer`

### 5.2 REST API (internal — web ↔ api)
See `docs/architecture.md` Section 4.2 for full endpoint list.

Request/response envelope:
```json
// Success
{ "data": { ... }, "meta": { "timestamp": "..." } }

// Error
{ "error": { "code": "INVALID_STEALTH_ADDRESS", "message": "..." } }
```

### 5.3 Database
- SQLite in development (file-based, zero-setup)
- PostgreSQL in production
- Access only via Prisma client — no raw SQL in application code

## 6. System Constraints

| Constraint | Detail |
|---|---|
| Bitcoin only (MVP) | ETH/token stealth payments are future work |
| BitGo testnet | All hackathon development uses Bitcoin Testnet |
| No hardware wallet | MVP does not support hardware signing devices |
| Single-user scanner | Scanner is per-wallet; multi-wallet fan-out is V2 |
| Node.js runtime | All backend services run on Node.js 20 (no Rust/Go services) |

## 7. Acceptance Criteria (Hackathon Demo)

| # | Scenario | Pass Condition |
|---|---|---|
| AC-01 | Receiver generates stealth address | QR code displayed; `(A, B)` exported as hex |
| AC-02 | Sender pays using stealth address | Tx broadcast; unique OTA per payment confirmed on-chain |
| AC-03 | Addresses are unlinkable | Two payments to same receiver appear at different on-chain addresses |
| AC-04 | Scanner detects payment | Within 2 scan cycles of tx confirmation, payment appears in receiver dashboard |
| AC-05 | Receiver spends payment | Spending tx broadcast successfully using derived `x` |
| AC-06 | Privacy guarantee | No correlation between stealth address and OTAs visible in block explorer |
